## 1. Abstract

Phase 4 adds **real-time voice capture and speech-to-text (ASR)** to the study bot. After the bot joins a voice channel via `/voicejoin`, it records all speaking participants in parallel using `discord.sinks.WaveSink`. Every 30 seconds a flush-and-restart cycle writes each speaker's audio buffer to a temporary WAV file, transcribes it locally using OpenAI Whisper (`base` model), and appends the labelled transcript to the active study session's `voice_transcript` list. The accumulated trancsript is incorporated into the end-of-session summary generated by the summariser at `/study end`.

A design priority throughout Phase 4 was **zero API cost for ASR** — Whisper runs entirely on CPU within the bot process, with the model loaded once per process lifetime via `@lru_cache` to avoid the 1–2 second model-load overhead on every transcription call.

## 2. Motivation & Research Context

In group study contexts, spoken discussion is a primary mode of knowledge consolidation. Traditional note-taking is disruptive; AI-powered transcription allows participants to focus on discussion while the bot silently captures the content for later summarisation, searchability, and topic reinforcement via the quiz engine.

Key technology choices:

- **OpenAI Whisper (local, open-source):** Whisper (Radford et al., 2022) is a robust, multilingual, noise-tolerant speech recognition model trained on 680,000 hours of web audio. The `base` model (74 M parameters) achieves near-SOTA WER on clean English audio and runs on CPU at roughly 2–4× real-time speed, making it practical for 30-second audio chunks with ~15–60 s processing latency.
- **discord.sinks (discord.py 2.x):** The `WaveSink` API introduced in discord.py 2.0's `VoiceClient` connects at the Opus RTP layer and demuxes each speaking user into a separate PCM buffer. This enables per-speaker attribution without any audio diarisation post-processing.
- **asyncio executor offloading:** Whisper's `model.transcribe()` is CPU-bound and synchronous. Wrapping it in `asyncio.get_event_loop().run_in_executor(None, ...)` offloads execution to a ThreadPoolExecutor thread, keeping the asyncio event loop unblocked for Discord heartbeats and command processing during the ~15–60 s transcription window.

## 3. Discord Server Setup Requirements

To use `/voicejoin` and `/voiceleave`, the following prerequisites must be met:

### 3.1 Bot Permissions

In the Discord server's channel permission settings for the target voice channel, the bot needs:

| Permission | Required For |
|---|---|
| **Connect** | Joining the voice channel |
| **Speak** | Activating the audio sink (discord.py requires this even for receive-only operation) |
| **Use Voice Activity** | Receiving per-user audio streams |

### 3.2 Gateway Intents

The `voice_states` gateway intent must be enabled. This is a **non-privileged** intent (no Developer Portal approval required) and is already included in the bot's `discord.Intents` configuration from Phase 1.

### 3.3 User Requirements

- The invoking user must already be in a voice channel at the time of `/voicejoin`. The bot joins the user's current channel.
- There is no requirement for other participants to be present at join time; they may join the channel after the bot.

### 3.4 ffmpeg Dependency

`openai-whisper` requires `ffmpeg` to be installed and available on `PATH` for audio decoding. On the development machine:

```bash
sudo apt install ffmpeg        # Debian/Ubuntu
brew install ffmpeg            # macOS
```

The `discord.sinks.WaveSink` output is 16-bit 48000 Hz PCM WAV, which Whisper can decode natively; however, ffmpeg is still required for Whisper's internal audio normalisation pipeline.

### 3.5 Voice Recording Consent Notice

Applicable data protection law (GDPR, CCPA) may require informed consent before recording participants. The recommended approach is to post a notice in the text channel when `/voicejoin` is invoked:

```python
await interaction.followup.send(
    "🔴 **Recording started.** All voice activity in this channel is being transcribed "
    "for the study session summary. Leave the channel at any time to opt out."
)
```

## 4. Phase 4 Architecture

```

  │
  ▼
cogs/voice.py  VoiceCog.join_voice()

  │  2. discord.VoiceClient.connect(channel)
  │  3. vc.start_recording(WaveSink(), _finished_recording, channel)
  │  4. asyncio.create_task(_chunk_flush_loop(guild_id, vc, channel))
  │
  ├─── every 30 seconds ──────────────────────────────────────────────
  │    _chunk_flush_loop
  │    5. vc.stop_recording() → triggers _finished_recording callback
  │    6. await asyncio.sleep(0.5)   # let callback complete
  │    7. vc.start_recording(WaveSink(), ...)   # fresh sink
  │
  ├─── on stop_recording ─────────────────────────────────────────────
  │    _finished_recording(sink, channel, *args)
  │    8. For each (member, audio_data) in sink.audio_data.items():
  │         a. Write audio_data.file bytes to NamedTemporaryFile (*.wav)

  │
  ▼
User calls /voiceleave
  │  9.  Cancel _chunk_flush_loop task
  │  10. vc.stop_recording()
  │  11. vc.disconnect()
  │
  ▼
At /study end (cogs/study.py)
  │  12. session["voice_transcript"] passed to generate_summary()
  └─ full spoken discussion captured in session summary
```

## 5. Component Deep-Dives

### 5.1 Model Loading — `@lru_cache` Singleton

```python
from functools import lru_cache
import whisper

@lru_cache(maxsize=1)
def _get_whisper_model():
    """Load the Whisper model exactly once for the process lifetime."""
    import whisper as _whisper
    model_name = config.WHISPER_MODEL   # default: "base"
    log.info(f"Loading Whisper model: {model_name}")
    return _whisper.load_model(model_name)
```

`@lru_cache(maxsize=1)` on a no-argument function creates a **process-lifetime singleton** — the model is loaded from disk on the first call and the returned object is cached for all subsequent calls. Without this, each 30-second flush cycle would re-load the ~140 MB `base` model weights, adding 1–2 seconds of I/O overhead per transcription batch.

`whisper.load_model()` is synchronous and CPU-bound. It is called during the first transcription cycle rather than at cog initialisation, so the bot starts up quickly and defers the one-time model-load cost to the first actual use.

### 5.2 Voice Connection — `join_voice()`

```python
@app_commands.command(name="voicejoin", description="Join your voice channel and start transcribing")
async def join_voice(self, interaction: discord.Interaction):
    await interaction.response.defer(thinking=True)

    # Validate: user must be in a voice channel
    if not interaction.user.voice or not interaction.user.voice.channel:
        return await interaction.followup.send(
            embed=embeds.error("You must be in a voice channel first.")
        )

    channel = interaction.user.voice.channel
    vc = await channel.connect()

    # Start WaveSink recording; callback fires on each stop_recording()
    vc.start_recording(
        discord.sinks.WaveSink(),
        self._finished_recording,
        interaction.channel,   # text channel for status messages
    )

    # Store in guild-scoped voice state
    self._voice_clients[interaction.guild_id] = vc
    self._flush_tasks[interaction.guild_id] = asyncio.create_task(
        self._chunk_flush_loop(interaction.guild_id, vc, interaction.channel)
    )

    await interaction.followup.send(embed=embeds.info(
        f"🎙️ Joined **{channel.name}**. Transcription running in 30s chunks."
    ))
```

### 5.3 Recording Sink — `WaveSink`

`discord.sinks.WaveSink` is a built-in discord.py sink that receives Opus RTP packets from the Discord voice server, decodes them to PCM, and buffers each speaker's audio in a separate `io.BytesIO` object (`sink.audio_data[member] → AudioData`). When `vc.stop_recording()` is called, the library finalises each buffer as a valid 16-bit 48000 Hz WAV file and calls the registered callback.

Key properties:
- **Per-speaker separation** is handled entirely by the library — no diarisation needed.
- `AudioData.file` is a `io.BytesIO` seeked to position 0, ready for writing.
- Encoding is `PCM_S16LE` at 48000 Hz, 2 channels (stereo). Whisper accepts this natively.

### 5.4 Incremental Flush Loop — `_chunk_flush_loop`

```python
CHUNK_INTERVAL_SECS = 30

async def _chunk_flush_loop(self, guild_id: int, vc: discord.VoiceClient,
                             channel: discord.TextChannel):
    try:
        while True:
            await asyncio.sleep(CHUNK_INTERVAL_SECS)

            if not vc.is_connected():
                break

            # Atomic stop → callback fires → restart recording
            vc.stop_recording()
            await asyncio.sleep(0.5)   # brief wait for callback to be scheduled

            if vc.is_connected():
                vc.start_recording(
                    discord.sinks.WaveSink(),
                    self._finished_recording,
                    channel,
                )

    except asyncio.CancelledError:
        pass   # clean cancellation on /voiceleave or bot shutdown
```

**Why 30 seconds?** This balances:
- **Latency to transcript:** transcript appears in `voice_transcript` every 30 s.
- **Whisper `base` model capacity:** 30 s of audio from multiple speakers is ~15–20 MB PCM, transcribed in ~30–60 s on CPU. Longer chunks would cause increasing lag where transcription is always "behind."
- **Discord API rate limits:** `stop_recording()` / `start_recording()` are not rate-limited, but excessive cycling wastes Discord connection overhead.

### 5.5 Transcription — `_transcribe_wav()`

```python
async def _finished_recording(self, sink: discord.sinks.WaveSink,
                               channel: discord.TextChannel, *args):
    session = self.bot.active_sessions.get(channel.guild.id)
    loop = asyncio.get_event_loop()

    for member, audio_data in sink.audio_data.items():
        speaker_name = member.display_name

        # Write buffer to a named temp file — Whisper requires a file path
        tmp_file = tempfile.NamedTemporaryFile(suffix=".wav", delete=False)
        try:
            tmp_file.write(audio_data.file.read())
            tmp_file.flush()
            tmp_path = tmp_file.name
        finally:
            tmp_file.close()

        try:
            # Offload CPU-bound transcription to thread pool
            result = await loop.run_in_executor(
                None,
                lambda p=tmp_path: _get_whisper_model().transcribe(p, fp16=False)
            )
            text = result["text"].strip()
            if text:
                line = f"[{speaker_name}]: {text}"
                if session:
                    session["voice_transcript"].append(line)
                log.info(f"Transcript: {line}")
        except Exception as e:
            log.warning(f"Whisper transcription failed for {speaker_name}: {e}")
        finally:
            # Always clean up temp file — even on transcription error
            try:
                os.remove(tmp_path)
            except FileNotFoundError:
                pass
```

**Critical implementation details:**

1. **`fp16=False`:** Whisper defaults to FP16 (half-precision) computation, which requires a CUDA-enabled GPU. On CPU, FP16 is not supported natively and causes a runtime error. Setting `fp16=False` forces FP32 computation, which is compatible with any host.

2. **`run_in_executor(None, ...)`:** Passes `None` as the executor, causing asyncio to use the default `ThreadPoolExecutor`. The lambda captures `tmp_path` by default argument (`p=tmp_path`) to avoid the common Python closure-capture loop variable bug.

3. **Temp file lifecycle:** The `finally` block runs `os.remove()` unconditionally — whether transcription succeeded, failed, or raised an exception. Without this, failed transcriptions would accumulate WAV files in `/tmp/`, eventually filling the disk.

4. **Empty text guard:** `if text:` prevents empty strings (silent segments, noise-only buffers) from polluting `voice_transcript`.

### 5.6 Disconnect — `leave_voice()`

```python
@app_commands.command(name="voiceleave", description="Stop transcribing and leave the voice channel")
async def leave_voice(self, interaction: discord.Interaction):
    await interaction.response.defer(thinking=True)

    task = self._flush_tasks.pop(interaction.guild_id, None)
    if task and not task.done():
        task.cancel()

    vc = self._voice_clients.pop(interaction.guild_id, None)
    if vc:
        vc.stop_recording()    # triggers one final _finished_recording callback
        await asyncio.sleep(0.5)
        await vc.disconnect()

    await interaction.followup.send(embed=embeds.info("👋 Left voice channel. Transcription stopped."))
```

The final `stop_recording()` before disconnect ensures the last (partial) 30-second audio chunk is transcribed and appended to `voice_transcript` rather than lost.

## 6. Whisper Model Selection

Whisper is available in five sizes. The `base` model is the default for this project:

| Model | Parameters | Disk | CPU Speed | WER (en) | Recommended Use |
|---|---|---|---|---|---|
| `tiny` | 39 M | ~75 MB | ~10× RT | High | Low-resource systems |
| **`base`** | **74 M** | **~140 MB** | **~5× RT** | **Medium** | **Default — this project** |
| `small` | 244 M | ~460 MB | ~2× RT | Lower | Better quality |
| `medium` | 769 M | ~1.5 GB | ~0.5× RT | Low | High-quality, fast CPU |
| `large` | 1550 M | ~3 GB | <real-time on CPU | Lowest | GPU only |

"RT" = real-time factor. "5× RT" means 30 s of audio transcribes in ~6 s on a modern CPU.

To switch models, update `config.py`:

```python
WHISPER_MODEL = "small"   # change to "tiny", "base", "small", "medium", or "large"
```

The `@lru_cache` singleton will load whichever model is configured at startup and hold it for the process lifetime.

## 7. CPU Performance Characteristics

Observed on the development server (AMD Ryzen 5 4500, 6-core, no AVX-512):

| Audio Duration | Model | Transcription Time | Real-Time Factor |
|---|---|---|---|
| 30 s | `base` | ~8–12 s | ~3–4× RT |
| 30 s × 3 speakers | `base` | ~25–40 s | serialised |
| 30 s | `small` | ~25–35 s | ~1× RT |

With the 30-second `CHUNK_INTERVAL_SECS`, transcription for 3 simultaneous speakers completes within the next flush cycle (~30 s), meaning the system is always approximately one cycle behind real-time.

If more than ~5 speakers are active simultaneously, the serialised transcription may not complete before the next flush. A Phase 6 improvement is to use a `ProcessPoolExecutor` for parallel per-speaker transcription.

## 8. Edge Cases & Failure Modes

### 8.1 User Joins Voice Channel After Recording Starts

`WaveSink` buffers audio per-speaker dynamically. New speakers joining after `start_recording()` are automatically included in subsequent `sink.audio_data` dictionaries without any code changes.

### 8.2 Short or Silent Audio Segments

If a speaker said nothing during a 30-second window, `audio_data.file.read()` returns a minimal WAV header with no PCM data. Whisper's `transcribe()` returns `{"text": ""}`. The `if text:` guard silently skips this.

### 8.3 `openai-whisper` Not Installed

If the import fails (e.g., package not installed), `_get_whisper_model()` raises `ImportError`. The `try/except Exception` in `_finished_recording()` catches this, logs a warning, and continues — no crash. The voice connection remains active and can receive audio; transcription is simply skipped.

### 8.4 Bot Disconnected Mid-Recording

If the bot is unexpectedly disconnected from the voice channel (server restart, network dropout), `vc.is_connected()` returns `False` in the next flush loop iteration, causing the loop to `break`. The in-progress `sink.audio_data` is lost and no final transcription occurs.

### 8.5 `/voicejoin` Without an Active Study Session

Voice transcription works independently of the study session:
- If a session is active, transcripts are appended to `session["voice_transcript"]`.
- If no session is active, the `if session:` guard skips the append silently. The transcript is logged but not stored.

### 8.6 Temp File Accumulation on Crash

If the process terminates during `os.remove(tmp_path)` — a highly unlikely but possible race — a stale `.wav` file may remain in `/tmp/`. These are typically cleaned up by the OS on next boot. A startup-time cleanup sweep of `/tmp/*.wav` could be added as a defensive measure.

## 9. Privacy & Consent Considerations

| Concern | Implementation | Status |
|---|---|---|
| Voice data persisted on disk | Raw WAV files deleted immediately after transcription in `finally` block | ✅ |
| Transcripts stored in DB | Transcripts are only in `session["voice_transcript"]` (in-memory) and included in the session summary, which is a text document | ✅ |
| User notified of recording | Informational embed posted to text channel on `/voicejoin` — opt-out by leaving VC | ⚠️ Recommended |
| Discord ToS compliance | Recording in voice channels is permitted under Discord's ToS when users are notified | ✅ |
| GDPR / CCPA | Users are UK/EU/US students; data minimisation satisfied (raw audio deleted, only text retained) | ✅ |

**Recommended addition for Phase 6:** `/voiceconsent` command that stores a "consent" flag in the `users` table. Only users who have run the consent command can have their audio transcribed.
